Model Compression Selection Guide

Choose a quantization algorithm, package, and GPU memory target for LLM deployment. The guide supports a blog article on compression and quantization approaches, with `llm-compressor` GPTQ as the production-oriented base path and practical alternatives for production, experiments, and edge deployment.

Default Recommendation

Start with `llm-compressor` unless your deployment target says otherwise. It is the vLLM-project compression toolkit, emits compressed-tensors metadata, and covers GPTQ, AWQ, RTN, SmoothQuant, AutoRound, FP8, KV-cache quantization, and newer FP4 paths.
Fit memory

Use GPTQ W4A16 with `llm-compressor` when the model does not fit comfortably in VRAM. Calibrate on a small representative dataset, quantize Linear layers, keep `lm_head` higher precision, then evaluate size, generation, perplexity, task metrics, and long-context behavior.

Throughput

Use dynamic FP8 W8A8 on Ada, Hopper, and newer GPUs when throughput is the goal and your runtime has the right kernels. FP8 usually beats int4 for high-concurrency serving because it avoids some weight-only dequantization overhead.

Experiments

Use bitsandbytes for fast local experiments and QLoRA-style fine-tuning. It is popular and easy, but it is runtime quantization rather than a clean vLLM production checkpoint export.

Alternative

Use GPTQModel when you want a current GPTQ/AWQ toolkit outside the `llm-compressor` workflow. Avoid starting new work on AutoAWQ or AutoGPTQ because their upstream repos now point users elsewhere.

Algorithm Matrix

Algorithm Package Best use Tradeoff
RTN W8A16 `llm-compressor` `QuantizationModifier` Fast baseline, little setup, low risk. Lower compression and usually lower quality recovery than GPTQ or AWQ.
GPTQ W4A16 `llm-compressor` `GPTQModifier`, GPTQModel Production int4 weight-only checkpoints when VRAM is the constraint. Needs calibration data and extra compression-time GPU memory for hessian matrices.
AWQ W4A16 `llm-compressor` `AWQModifier` plus `QuantizationModifier` Int4 weight-only deployment, often strong for instruction-tuned and multimodal models. Model family mappings matter. Validate before rollout.
SmoothQuant W8A8 `llm-compressor` Activation quantization with outlier smoothing on broad GPU support. More moving parts than weight-only quantization.
Dynamic FP8 `llm-compressor`, NVIDIA ModelOpt High-throughput serving on Ada, Hopper, and newer GPUs. Hardware and runtime kernel support decide whether it is available.
KV cache quantization vLLM, selected compressed checkpoints Long-context and high-concurrency serving where cache memory dominates. Can affect quality. Test retrieval and long-context tasks.
bitsandbytes LLM.int8 / NF4 Transformers + bitsandbytes + PEFT Experiments, local inference, and QLoRA fine-tuning. Less suitable as a vLLM compressed checkpoint publishing path.
GGUF K-quants llama.cpp, Ollama, MLX-LM CPU, Apple Silicon, desktop, and edge deployments. Different runtime ecosystem from vLLM server GPU deployment.

Package Choice

Package Choose it when Do not choose it when
`llm-compressor` You want a production checkpoint for vLLM, compressed-tensors metadata, GPTQ/AWQ/FP8, and an auditable offline compression workflow. You only need quick local runtime quantization or Apple/CPU edge deployment.
bitsandbytes You want simple `Transformers` runtime loading, LLM.int8, NF4, or QLoRA fine-tuning. You need a portable vLLM artifact with an offline compression audit trail.
GPTQModel You want a current alternative for GPTQ/AWQ across Transformers, Optimum, PEFT, vLLM, and SGLang. You are already committed to the vLLM project's `llm-compressor` flow.
llama.cpp / GGUF You deploy on CPU, Apple Silicon, mobile, desktop, or edge runtimes. You need high-concurrency vLLM serving with continuous batching.
TensorRT-LLM / NVIDIA ModelOpt You are standardizing on NVIDIA's deployment stack and can tune per GPU generation. You need the simpler open vLLM path or non-NVIDIA portability.

Hardware and Memory Rules

Weights
1B fp16/bf16 parameters are roughly 2 GB before overhead. W4A16 only quantizes selected weights, so the real model reduction is less than the theoretical 4x.
KV cache
KV memory scales with layers, hidden dimension, context length, concurrency, and KV precision. Long context can erase the savings from weight quantization.
Compression job
Serving fit and compression fit are different. GPTQ needs auxiliary memory during calibration; `llm-compressor` text decoding flows commonly onload one layer at a time.

Recommended Workflow

1. Baseline

Run bf16/fp16 generation and task evals. Save latency, throughput, memory, and quality numbers.

2. Compress

Start with GPTQ W4A16 for fit or FP8 dynamic for throughput on modern GPUs.

3. Validate

Compare generation samples, perplexity, task metrics, and long-context behavior.

4. Serve

Load in vLLM or the target runtime with the real context and concurrency profile.

5. Decide

Keep the smallest format that preserves quality and meets latency, throughput, and cost goals.

GPTQ compression command

uv run python demo.py quantize \
  --algorithm gptq-w4a16 \
  --model Qwen/Qwen3-0.6B \
  --output-dir outputs/Qwen3-0.6B-W4A16 \
  --dataset wikitext \
  --dataset-config-name wikitext-2-raw-v1 \
  --num-calibration-samples 256 \
  --max-seq-length 4096

Planner command

uv run python demo.py plan --params-b 7 --goal fit-memory --hardware ampere --context 4096 --concurrency 4

Quality evaluation command

uv run python demo.py quality-eval \
  --base-model Qwen/Qwen3-0.6B \
  --compressed-model outputs/Qwen3-0.6B-W4A16 \
  --mode all \
  --lm-eval-task hellaswag \
  --output-json reports/qwen3-0.6b-w4a16-quality.json
